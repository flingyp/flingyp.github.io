在日常使用中，我的对于模型能力绝大部分时间都是让它实现自己已经明确清晰化的需求。所以我在使用模型时，更加会在乎的是它生成的代码是否靠谱。

## Claude Code

> 已知，明确的

日常使用 Claude Code 频率是最高的。包括帮助去理解代码，梳理逻辑，编写文档以及编写日常开发需求，基本都是通过 Claude Code 来实现的。

下面这个是我日常需求开发使用会用到的需求开发模版提示词文档。

```markdown
## Feature

## Documentation

## Tips

- 修改过的文件中添加注释 `TODO: ${需求名称}-开发中` 字样
  - development 和 logs 文件夹下文件除外
- 功能实现如依赖第三方插件，请使用 PNPM 包管理工具
- 无需过多考虑功能的完整性
  - 如果需要额外添加需求未提到的功能，请先询问我获取我的同意，在进行扩展
- 无需修复导入顺序和类型导入的问题
  - 在修复其问题如果出现长期无法修复的问题，需要在文件中标志注释 `TODO: 类型问题-待修复` 即可
  - 然后跳过类型修复问题
- 需求功能开发后后都需要在当前标题标注【已完成】
- 最后需求完成后，总结一份日志报告
  - 要整体描述需求/BUG，以及需求功能的实现或BUG修复内容，以及修改的文件路径和描述具体改了什么
  - 输出报告至 `logs` 文件夹中，文件名为：`${YYYYMMDD}-${需求名称}.md`
```

## 个人感悟

1. AI 时代情况下，代码规范和质量也是同等重要的
2. 项目本身就是屎山的情况下，不要怪 AI 写出来代码屎山
3. AI 是遇强则强，遇弱则弱
4. 不要指望 AI 干活能够一步到位
5. 改变自己和 AI 的定位，把自己当作指挥官，AI 当作你的打手。打手干的好不好，最终还是依赖于指挥官指挥的怎么样

### 下面是 AI 帮忙优化过一轮的

AI写代码不是万能钥匙，真正的关键在你自己

1. 在 AI 时代，代码规范和质量依然同样重要。AI 再聪明，也无法弥补一个混乱的项目结构。
2. 如果项目本身就是屎山，那 AI 写出来的自然也不会变黄金。垃圾输入，只会有垃圾输出。
3. AI 是“遇强则强，遇弱则弱”。你越懂技术、越懂需求，AI 的输出就越惊艳。
4. 不要幻想 AI 一键帮你搞定所有事。它是工具，不是魔法。
5. 真正的心法：把自己当指挥官，AI 当你的打手。
